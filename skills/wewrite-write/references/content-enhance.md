# 内容增强策略

增强的目标是让主张更值得读、更能被证据支撑，不是强行反转、堆数字或制造故事。先完成
`brief.yaml` 与 `claims.yaml`，再按框架选择一项主策略；材料不够时缩小主张或换框架。

## 观点 / 热点解读：可支持的新角度

先浏览代表性的已有解释，列出共识、分歧和被忽略的条件。从中选择一个有证据的新贡献：
更准确的因果解释、更清楚的适用边界、对具体读者的影响，或更好用的判断框架。不能为了
“尖锐”反对主流，也不能把推测写成内幕。输出到 `thesis.novelty` 和对应主张。

## 痛点 / 清单：行动可用性

逐节检查读者能否开始行动，并补齐必要的步骤、条件、成本、风险和失败信号。具体工具、
参数、价格和版本都属于事实，必须核实且注明时效。不是每节都需要数字；不能把泛泛建议
包装成确定答案。输出到 `goal.action`、章节目的和主张边界。

## 故事 / 复盘：有来源的真实细节

只使用用户明确提供或可靠来源明确记载的时间、地点、对话和细节。直接引语必须能逐字核对；
转述要标明来源。材料不足时改用事实叙述或换成观点/复盘框架，禁止“合理重建”、拼接人物、
添加感官细节或替用户补经历。输出到 `personal_materials`、来源账本和主张清单。

## 对比：决策条件

先定义共同标准，再比较证据、限制、总成本和适用人群。官方信息用于功能和价格，用户评价
只代表被引用者的体验，不能外推为普遍事实。无法验证的“隐藏坑”不写。输出一个清楚的
“什么情况下选什么”，而不是整齐但无结论的优缺点表。

## 通过检查

- 新角度能由已有证据和逻辑支持，而不是为了不同而不同。
- 具体信息都有来源或明确边界。
- 没有把第三方故事、范文片段或模型记忆改写成作者亲历。
- 每个增强点都服务读者问题和核心主张；删掉后毫无影响的内容不加入。
